# 1. Introduction: when you want a friendly time-series model

By now you've met:

- ARIMA: the classic, slightly formal time series friend,
- ARMA: the "I know all your autocorrelations" cousin.

Enter **Prophet**, which is like the colleague who says:

> "Don't worry, I'll handle the trend, seasonality, and holidays for you - just give me a data frame."

Prophet was designed with a "data scientist first" mindset:

- You give it a time series with a date column,
- It decomposes the series into **trend + seasonality + holidays**,
- It produces a forecast with intervals and nice plots,
- You don't have to wrestle with ACF/PACF unless you want to.

Is it magic? No. Is it convenient and often quite good? Yes.

In this tutorial we'll:

- Lay out the main ideas behind Prophet,
- Fit a Prophet model in R on synthetic "hospital admissions" data,
- Compare Prophet with ARIMA in terms of assumptions, workflow, and flexibility,
- List 4 strengths and 4 limitations,
- Explain why Prophet-style models matter for **HEOR and health policy**,
- Point to some further reading (including critical perspectives).

---

# 2. Foundations: what is Prophet doing?

Prophet models a time series $y(t)$ as a sum of components:

$$
y(t) = g(t) + s(t) + h(t) + \varepsilon_t,
$$

where:

- $g(t)$ is a **trend** term,
- $s(t)$ captures **seasonality** (weekly, yearly, etc.),
- $h(t)$ captures **holiday / special event effects**,
- $\varepsilon_t$ is an error term.

## 2.1. Trend

The trend $g(t)$ is typically modeled as a piecewise linear or logistic growth curve, with:

- **Change points** where the slope can shift,
- Priors that control how flexible the trend is.

This is helpful when:

- There are changes in growth rate (e.g., policy introduction, new program),
- The series doesn't follow a simple straight or gently curved line.

## 2.2. Seasonality

Prophet uses **Fourier series** to model seasonality:

- Annual seasonality (e.g., winter peaks),
- Weekly seasonality (e.g., Monday vs Sunday),
- Additional seasonalities (e.g., monthly, if needed).

These are smooth periodic components added to the model.

## 2.3. Holidays and special events

You can pass a data frame with **holiday dates** (and possibly regions), and Prophet will estimate:

- Effects for those dates,
- Potentially spillover effects for days around them.

For HEOR/health policy, "holidays" could also represent:

- Special campaigns,
- Shocks,
- Implementation dates for big reforms,
- Pandemics (though those are more than just "holidays" 😅).

## 2.4. Prophet as a generalized additive model (GAM)

Under the hood, Prophet is a kind of **additive model** with:

- Parametric trend,
- Smooth seasonal components,
- Optional holiday regressors.

You can think of it as:

> A GAM that's pre-configured to be extremely convenient for common forecasting tasks, especially daily or monthly data.

# 3. Example in R: Prophet on synthetic "hospital admissions"

We'll simulate **monthly hospital admissions** over 10 years:

- Slight increasing trend,
- Strong yearly seasonality,
- Some random noise.

Then we'll:

- Fit ARIMA (for reference),
- Fit Prophet,
- Plot both forecasts conceptually (using Prophet's tools),
- Compare them qualitatively.

> Note: Prophet works best with **date-time** columns named `ds` (date) and `y` (value).

```r

set.seed(123)

# Simulate 10 years of monthly data
n_years  <- 10
freq     <- 12
n_period <- n_years * freq

dates <- seq.Date(
  from = as.Date("2010-01-01"),
  by   = "month",
  length.out = n_period
)

time_index <- 1:n_period

# Components for synthetic admissions:
baseline   <- 100
trend      <- 0.7 * time_index             # slow upward trend
seasonality <- 12 * sin(2 * pi * time_index / freq)  # yearly cycle
noise       <- rnorm(n_period, mean = 0, sd = 8)

admissions <- baseline + trend + seasonality + noise

# Data frame for Prophet
library(dplyr)

df <- tibble::tibble(
  ds = dates,
  y  = admissions
)

head(df)
```

---

## 3.1. Train-test split

We'll use:

- First 8 years as **training**,
- Last 2 years as **test**.

```r

train_end_date <- as.Date("2017-12-01")  # 8 years from 2010-01-01
train_df <- df %>% dplyr::filter(ds <= train_end_date)
test_df  <- df %>% dplyr::filter(ds >  train_end_date)

nrow(train_df); nrow(test_df)
```

---

## 3.2. Fitting ARIMA as a baseline

We'll fit ARIMA using the `forecast` package on the same training period.

```r

library(forecast)

train_ts <- ts(train_df$y, frequency = freq)

fit_arima <- auto.arima(train_ts)
fit_arima
```

We'll generate forecasts for the test horizon:

```r

h <- nrow(test_df)
fc_arima <- forecast(fit_arima, h = h)

# Extract ARIMA predictions as a data frame aligned with test dates
arima_pred <- tibble::tibble(
  ds = test_df$ds,
  yhat_arima = as.numeric(fc_arima$mean)
)
```

---

## 3.3. Fitting Prophet

Now we fit a Prophet model on the same training data.

```r

# install.packages("prophet") # run once if not installed
library(prophet)

m <- prophet(train_df)

# Create future dataframe for the full horizon (train + test period)
future <- make_future_dataframe(m, periods = nrow(test_df), freq = "month")
future
```

Now we get the forecast:

```r

forecast_prophet <- predict(m, future)

# Extract only the test-period predictions and keep relevant columns
prophet_pred <- forecast_prophet %>%
  dplyr::filter(ds > train_end_date) %>%
  dplyr::select(ds, yhat, yhat_lower, yhat_upper)
```

---

## 3.4. Comparing Prophet and ARIMA forecasts

We'll combine:

- Actual test values,
- ARIMA predictions,
- Prophet predictions.

```r

compare_df <- test_df %>%
  dplyr::left_join(arima_pred,   by = "ds") %>%
  dplyr::left_join(prophet_pred, by = "ds")

head(compare_df)
```

Let's visualize them:

```r

library(ggplot2)

ggplot(compare_df, aes(x = ds)) +
  geom_line(aes(y = y,            color = "Actual")) +
  geom_line(aes(y = yhat_arima,   color = "ARIMA")) +
  geom_line(aes(y = yhat,         color = "Prophet")) +
  geom_ribbon(
    aes(ymin = yhat_lower, ymax = yhat_upper),
    alpha = 0.2
  ) +
  labs(
    x = "Date",
    y = "Admissions",
    title = "Prophet vs ARIMA Forecasts (Test Period)"
  ) +
  scale_color_manual(values = c("Actual" = "black", "ARIMA" = "blue", "Prophet" = "red"))
```

We can compute simple metrics:

```r

# Remove any rows with missing values (if any)
compare_df_clean <- na.omit(compare_df)

true_vals   <- compare_df_clean$y
arima_vals  <- compare_df_clean$yhat_arima
prophet_vals <- compare_df_clean$yhat

mae <- function(a, b) mean(abs(a - b))
rmse <- function(a, b) sqrt(mean((a - b)^2))

c(
  MAE_ARIMA   = mae(true_vals, arima_vals),
  RMSE_ARIMA  = rmse(true_vals, arima_vals),
  MAE_Prophet = mae(true_vals, prophet_vals),
  RMSE_Prophet = rmse(true_vals, prophet_vals)
)
```

This gives a sense of relative performance in this particular simulated example.
In real HEOR data, you'd repeat this over multiple series or use rolling-origin
evaluation.

# 4. Strengths and limitations of Prophet

## 4.1. Four strengths

1. **Very user-friendly interface**  
   You give it `ds` and `y`, get a model and forecast with trend + seasonality + holidays, and a plot. Great for quick applied work and exploratory forecasting.

2. **Handles trend changes and seasonality well**  
   Prophet is particularly good when:

   - There are clear seasonal patterns (yearly, weekly),
   - Trends change at a few points,
   - You have daily or monthly granular data.

3. **Built-in holiday / special event effects**  
   You can easily specify "holiday" dates (or pseudo-holidays like major policy launches) and let the model estimate their effects.

4. **Interpretable decomposition**  
   Prophet's decomposition into trend, seasonality, and holidays is easy to visualize and explain to non-technical stakeholders, which is valuable in policy and HEOR settings.

---

## 4.2. Four limitations

1. **Less flexible for complex autocorrelation structures**  
   Prophet focuses on trend + seasonality + holidays, but does not explicitly model short-range autocorrelation the way ARMA does. Residual autocorrelation can remain.

2. **Defaults can be misleading if not checked**  
   Because Prophet is so "automatic," it can be tempting to accept defaults. But:

   - Trend priors might be too flexible or too rigid,
   - Seasonal components might underfit or overfit,
   - Diagnostics are still needed.

3. **Not always the best for short series or weak seasonality**  
   When there is:

   - Very little data,
   - No clear seasonality,
   - Unstable patterns,

   simpler models (or ARIMA) can perform as well or better, with less complexity.

4. **Model assumptions may not match all HEOR use cases**  
   Prophet's structure is tuned for typical business/website time series (daily counts, clear seasonal patterns). Some HEOR data (e.g., irregular measurements, strong interventions, structural breaks) may require more custom modeling.

# 5. Prophet vs ARIMA: a quick comparison

Here's a rough qualitative comparison to help you decide when each might shine.

## 5.1. Modeling philosophy

- **ARIMA**:
  - Focuses on **autoregression, differencing, and moving averages**.
  - Often assumes stationarity after differencing.
  - Captures short-term dynamics very explicitly.

- **Prophet**:
  - Treats the series as **trend + seasonality + holidays**.
  - Uses change points and smooth seasonal terms.
  - Emphasizes interpretable decomposition and easy configuration.

## 5.2. Practical workflow

- **ARIMA**:
  - You may look at ACF/PACF, try different orders, check residuals.
  - `auto.arima()` helps, but some understanding of ARIMA structure is helpful.
  - Seasonality is handled via SARIMA terms.

- **Prophet**:
  - You pass a data frame with `ds` and `y`.
  - You optionally add holidays/extra regressors.
  - You rely heavily on defaults, with tuning via priors and seasonalities.

## 5.3. Data patterns

- **ARIMA** tends to work well when:
  - The series is reasonably stationary after transformations,
  - Dynamics are dominated by short memory and simple seasonality,
  - You have a moderate length of history.

- **Prophet** tends to work well when:
  - There is a strong, smooth trend,
  - Clear, recurring seasonal patterns (e.g., yearly, weekly),
  - Potentially important holidays or events.

## 5.4. Communication and interpretation

- **ARIMA**:
  - Well-known in statistical time series literature,
  - Parameters have interpretation in terms of lags and shocks,
  - Less visually decomposed by default.

- **Prophet**:
  - Decomposition plots (trend, yearly seasonality, weekly seasonality, holidays) are very intuitive,
  - Easier to explain to non-technical audiences: "this is seasonal pattern over the year, here's the long-term trend, here are holiday bumps."

In practice, an HEOR analyst might:

- Use ARIMA and Prophet in parallel,
- Compare forecasts via rolling-origin validation,
- Choose based on performance + interpretability + ease of use.

# 6. Why Prophet-style models matter for HEOR and health policy

Prophet (and similar additive time series models) can be especially helpful in HEOR and policy work because they:

## 6.1. Provide quick, interpretable forecasts

For questions like:

- "What will monthly admissions look like over the next 2-3 years?"
- "What's the likely range of drug utilization if seasonal patterns continue?"

Prophet provides:

- Forecasts with intervals,
- Trend and seasonality plots,
- Fast turnaround for scenario discussions.

## 6.2. Highlight seasonal and holiday effects

Seasonality and special events matter for:

- **Capacity planning** (e.g., winter peaks, holiday dips),
- **Staffing**,
- **Supply chain management**.

Prophet's decomposition makes it easy to show:

- "Here are winter peaks,"
- "Here is the effect of holidays,"
- "Here is the underlying trend."

## 6.3. Serve as baselines for policy evaluation

You can use Prophet to:

- Model pre-policy trends and expected seasonal patterns,
- Forecast a **counterfactual** trajectory (what would have happened without the policy),
- Compare observed post-policy series to the forecasted baseline.

While not a substitute for a full causal design (DiD, ITS with more structure, etc.), it can provide:

- A transparent baseline,
- A way to communicate expected vs observed trajectories.

## 6.4. Integrate into broader decision models

Prophet forecasts (and seasonal decompositions) can feed into:

- Budget impact models,
- Capacity and resource planning tools,
- Simulation models that require time-varying inputs.

Because Prophet returns a full distribution of forecasts, you can also:

- Sample from forecast distributions,
- Propagate uncertainty into downstream models.

# 7. Further reading

If you'd like to learn more about Prophet and how it fits into the time series landscape (including pros/cons), here are four useful resources:

1. **Taylor & Letham - _Forecasting at Scale_ (original Prophet paper)**  
   Describes the motivation, model structure, and application of Prophet to business time series.

2. **Prophet documentation / tutorials (R & Python)**  
   Walkthroughs of trend/seasonality/holiday modeling, extra regressors, and diagnostics. Great for hands-on learning.

3. **Hyndman & Athanasopoulos - _Forecasting: Principles and Practice_**  
   Not Prophet-specific, but an excellent free resource for understanding time series decomposition, ARIMA, ETS, and model evaluation - essential context for interpreting Prophet.

4. **Blog posts and critiques comparing Prophet to ARIMA/ETS**  
   Several forecasting experts have written about when Prophet works well and when classical methods may be preferable; reading a mix of views helps put Prophet in perspective.

With Prophet added to your toolkit, you can now:

- Let ARIMA handle the "pure time-series-statistics" side,
- Let Prophet handle fast, interpretable trend/seasonality forecasting,
- And decide, case-by-case, which tool makes more sense for the HEOR and policy story you want to tell. 😄
